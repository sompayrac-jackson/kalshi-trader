## What Went Wrong: Two Critical Bugs

### Bug 1 — Future Game Ticker Contamination (PRIMARY CAUSE)

**The scanner is entering future baseball games using live game data.**

Root cause: `scan_live()` fetches ALL open Kalshi MLB markets with no date filter. This includes markets for May 27, 28, and 29 games. The `baseball_signal()` function then matches the Kalshi market's team name against any live ESPN game, regardless of whether the dates match.

**Example — KXMLBGAME-26MAY291845SDWSH-WSH at 22:56 UTC:**
1. Live ESPN game: WSH vs CLE, May 26, winning in late innings → model says WSH 73%
2. Kalshi ticker: SD vs WSH, May **29** game — currently priced at 17c (pre-game)
3. Scanner applies live WSH win probability (73%) to future MAY29 ticker priced at 17c
4. Computed edge = 0.73 − 0.17 = **+0.56** (appears to be a massive opportunity)
5. Buys 6 contracts at 73c cost
6. Immediately: SL fires because bid = 17c (that is the actual market price, -76.7% drop)
7. Next cycle: ESPN still shows live WSH game → scanner re-enters MAY29 ticker → SL fires again

This has been happening since **May 24**. From the orders log, affected tickers include:
- `26MAY27` tickers: dozens of entries on May 24 and 25
- `26MAY28` tickers: same pattern
- `26MAY29` tickers: entries starting May 26 evening

The fix is a **single line in `live_scanner.py`**: parse the game date from the ticker string and only include markets where the game date is today or yesterday (for late-night games crossing UTC midnight).

---

### Bug 2 — Re-Entry Cascade After Stop-Loss (SECONDARY CAUSE)

**After a SL exit, the scanner re-enters the same position within 30 seconds.**

Sequence:
1. SL fires → `execute_exit()` removes ticker from `_open_tickers` → exit logged
2. **Same scan cycle**: scanner calls `scan_live()` BEFORE `_check_exits()` (see scanner loop order). But even across cycles:
3. **Next cycle (32 seconds later)**: ticker is no longer in `_open_tickers`, so `execute_live()` re-enters
4. `_open_tickers` is updated with the new position
5. Price hasn't recovered → exit check fires → SL hits again → removes from `_open_tickers`
6. Repeat forever

**Observed today:**
- WSHCLE-WSH: exited SL, re-entered 31 seconds later, looped **48 more times**
- COLLAD-COL: 87 total exit fires across multiple sessions
- SDWSH-WSH: exited, re-entered immediately, looped 6 times
- MINPIT-PIT, LAAATB-LAA: 3-6 fires each

The fix is a **30–60 minute SL cooldown**: after any stop-loss exit, block re-entry on the same ticker for a configurable period.

## Stop-Loss Slippage Problem

Many SL exits show massive slippage: entry 70–73¢, exit 17–25¢. This is a 65–76% price drop at exit vs a 35% SL threshold.

**Why this happens:**
- SL triggers when bid drops to ~47¢ (35% below 73¢ entry)
- But MLB markets are illiquid — once a game turns decisively, bids jump from 73¢ directly to 15–20¢ with no intermediate liquidity
- By the time the SL is detected and the sell order placed, the market is already at 15–25¢
- The stop-loss threshold fires correctly, but the fill is far below the trigger price

**Implication:** The SL doesn't protect the 35% threshold — it's the best available exit price at the time. This is not fixable with a threshold change. The only real solutions are:
1. Don't enter high-probability (70–73¢) positions where a loss is catastrophic (each lost contract loses nearly the full entry price)
2. Accept that illiquid markets have wide bid slippage and size accordingly

## Priority Action Items

### Critical — code changes needed now

#### 1. Fix future game ticker contamination (`live_scanner.py`)

In `scan_live()`, replace the baseball markets query with a date-filtered version:

```python
from datetime import date, timedelta

def _ticker_game_date(ticker: str) -> date | None:
    import re
    months = {'JAN':1,'FEB':2,'MAR':3,'APR':4,'MAY':5,'JUN':6,
              'JUL':7,'AUG':8,'SEP':9,'OCT':10,'NOV':11,'DEC':12}
    m = re.search(r'(\d{2})(JAN|FEB|MAR|APR|MAY|JUN|JUL|AUG|SEP|OCT|NOV|DEC)(\d{2})', ticker)
    if not m: return None
    return date(2000 + int(m.group(1)), months[m.group(2)], int(m.group(3)))

today = now.date()
yesterday = today - timedelta(days=1)
baseball_markets = [
    m for m in client._get("/markets", params={"limit": 100, "series_ticker": "KXMLBGAME", "status": "open"}).get("markets", [])
    if not m.get("result")
    and m.get("yes_ask_dollars")
    and _ticker_game_date(m.get("ticker", "")) in (today, yesterday)
]
```

This eliminates MAY27/28/29 entries entirely.

#### 2. Add SL cooldown to prevent re-entry (`order_executor.py` + `dashboard.py`)

In `OrderExecutor`:
```python
self._sl_cooldown: dict[str, float] = {}  # ticker → timestamp of SL exit
SL_COOLDOWN_SEC = 3600  # 60 minutes, make configurable later
```

In `execute_live()` before placing any order:
```python
sl_ts = self._sl_cooldown.get(ticker, 0)
if time.time() - sl_ts < SL_COOLDOWN_SEC:
    return ExecResult("skipped", 0, 0, "SL cooldown active — re-entry blocked")
```

In `execute_exit()` when `reason == "stop_loss"`:
```python
self._sl_cooldown[ticker] = time.time()
```

#### 3. Move `_check_exits` before `scan_live` in scanner loop (`dashboard.py`)

Current order:
1. `scan_live` (may re-enter just-exited position)
2. fetch positions
3. `_check_exits`

Better order:
1. fetch positions
2. `_check_exits`
3. `scan_live`

This prevents buying a position and immediately stopping it out in the same cycle. Won't fully fix the cascade (that requires the cooldown), but reduces the damage window.
